# ADR-0012: Centralized Immutable State Store

**Status:** Accepted
**Date:** 2026-04-01
**Domain:** State Management
**Context:** `src/state/AppStateStore.ts` (569 lines), `src/state/store.ts`

## Context

The application needs a single source of truth for state that is predictable, debuggable, and type-safe.

## Decision

Implement a centralized immutable state store with:
- `DeepImmutable<AppState>` wrapper for type safety
- `setAppState(f: (prev) => prev)` pattern for updates
- Selectors for derived state
- Redux-like store implementation

## Consequences

### Positive
- Predictable state transitions
- Type-safe state access
- Easy debugging and time-travel
- Single source of truth

### Negative
- Large AppState type (569 lines)
- Immutable updates can be verbose
- Performance overhead for deep immutability

# ADR-0013: React Context for UI State

**Status:** Accepted
**Date:** 2026-04-01
**Domain:** State Management
**Context:** `src/context/` (9 providers)

## Context

UI components need access to specific state slices without prop drilling.

## Decision

Use React context providers for:
- Voice state
- Stats
- Notifications
- Modal state
- Mailbox
- And other UI-specific state

## Consequences

### Positive
- Clean component hierarchy
- Targeted re-renders
- Separation from global store

### Negative
- Multiple context providers increase complexity
- Context re-render optimization required
