# ADR-0014: Multi-Layer Permission System

**Status:** Accepted
**Date:** 2026-04-01
**Domain:** Permissions/Security
**Context:** `src/types/permissions.ts`, `src/utils/permissions/`

## Context

AI-driven code execution requires granular permission control to prevent unauthorized or destructive operations.

## Decision

Implement a multi-layer permission model:

**Permission Modes:**
- `default` - Standard permission flow
- `plan` - Planning mode
- `acceptEdits` - Auto-accept edits
- `bypassPermissions` - Skip permission checks
- `dontAsk` - Never ask for permission
- `auto` - Automatic decisions
- `bubble` - Bubble up to parent

**Permission Rules:**
- `alwaysAllow` - Auto-approve
- `alwaysDeny` - Auto-reject
- `alwaysAsk` - Always prompt user

**Rule Sources:** user, project, local, policy, CLI, command, session

## Consequences

### Positive
- Granular control over tool execution
- Flexible rule sources
- Mode-based permission behavior
- Security by design

### Negative
- Complex permission resolution logic
- Multiple sources can conflict
- User experience complexity

# ADR-0015: YOLO Auto-Approval Classifier

**Status:** Accepted
**Date:** 2026-04-01
**Domain:** Permissions/Security
**Context:** `src/utils/permissions/`

## Context

Frequently running safe commands (like `ls`, `cat`) requires user interaction for each execution, reducing productivity.

## Decision

Implement a YOLO classifier that automatically approves safe commands without user interaction.

## Consequences

### Positive
- Improved productivity for safe operations
- Reduced user friction
- Maintains security for dangerous operations

### Negative
- Classifier false positives could be dangerous
- Classifier logic must be carefully maintained
- Security boundary depends on classifier accuracy
