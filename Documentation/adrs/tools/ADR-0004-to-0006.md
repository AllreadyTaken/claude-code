# ADR-0004: Polymorphic Tool Interface

**Status:** Accepted
**Date:** 2026-04-01
**Domain:** Tools
**Context:** `src/Tool.ts` (792 lines), `src/tools/`

## Context

The AI model needs to interact with the external world (file system, shell, web, etc.) through a unified interface that supports validation, permissions, execution, and UI rendering.

## Decision

Define a rich `Tool` interface with:
- `call()` - execution
- `description()` - dynamic description generation
- `inputSchema` / `inputJSONSchema` - Zod or JSON Schema validation
- `checkPermissions()` - permission checking
- `renderToolUseMessage()` / `renderToolResultMessage()` - UI rendering
- `isConcurrencySafe()`, `isReadOnly()`, `isDestructive()` - safety metadata
- `buildTool()` factory with sensible defaults

## Consequences

### Positive
- Uniform interface for all 43 tools
- Built-in permission checking
- Consistent validation via Zod/JSON Schema
- Safety metadata for concurrency and destructiveness
- Factory pattern reduces boilerplate

### Negative
- Large interface (792 lines)
- Tools must implement many methods
- Tight coupling between tool logic and UI rendering

# ADR-0005: Tool Registry and Assembly

**Status:** Accepted
**Date:** 2026-04-01
**Domain:** Tools
**Context:** `src/tools.ts`

## Context

Tools need to be dynamically assembled based on context, permissions, and feature flags.

## Decision

Centralize tool management in `tools.ts` with registry, assembly, and filtering logic. Tools are filtered based on:
- Feature flags
- Permission modes
- Client type
- Session configuration

## Consequences

### Positive
- Single point of tool management
- Dynamic tool availability
- Feature flag integration

### Negative
- Central file becomes a bottleneck
- Complex filtering logic

# ADR-0006: Tool Safety Metadata

**Status:** Accepted
**Date:** 2026-04-01
**Domain:** Tools
**Context:** `src/Tool.ts`, `src/tools/`

## Context

Tools have different safety profiles - some are read-only, some are destructive, some can run concurrently.

## Decision

Each tool declares safety metadata:
- `isConcurrencySafe()` - can run in parallel
- `isReadOnly()` - no side effects
- `isDestructive()` - modifies external state

## Consequences

### Positive
- Automatic permission decisions based on safety
- Safe parallel execution
- Clear tool contracts

### Negative
- Metadata must be maintained per tool
- Incorrect metadata can cause security issues
